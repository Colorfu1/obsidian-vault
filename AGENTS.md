# Obsidian Vault Repository Instructions

本仓库是一个以中文技术笔记为主的 Obsidian 知识库。修改笔记时，不只维护单个 Markdown 文件，还要维护文章之间的双链、主题索引、阅读状态和 GitHub 导航。

## 开始工作前

1. 先运行 `git status -sb`，确认用户本轮新增、修改、删除或重命名了哪些文件。
2. 不要撤销或覆盖用户已有改动。重命名文件应视为原文章的延续，不要当成无关的新文件重新生成。
3. 依次阅读 `README_for_GPT.md`、`index/master_index.md` 和最相关的主题索引，再判断文章应放入哪个目录及与哪些笔记建立联系。
4. 优先搜索整个仓库后再编辑：文件搜索使用 `rg --files`，内容搜索使用 `rg -n`。

## 目录规则

- `Robot/AWM/`：action/world model、视频预测、model-based RL、潜空间规划等笔记。
- `Robot/VLA/PI/`：Physical Intelligence 系列及其 action token、memory、policy improvement 笔记。
- `Robot/VLA/`：其他 VLA、机器人基础模型、模仿学习、动作生成和机器人硬件笔记。
- `RL/`：强化学习、策略优化、蒸馏和相关概念笔记。
- 仓库根目录：明确跨主题、且不适合归入现有领域目录的基础概念笔记。
- `index/`：导航和主题索引，不放长篇正文。

若新主题无法自然归入现有目录，先沿用最接近的目录；只有在同类文章已经形成稳定类别时才创建新目录和主题索引。

## 文件命名

- 文件名应直接描述论文、模型或概念，例如 `RDT-1B.md`、`PPO.md`、`Dreamer技术报告.md`。
- 不要在文件名、标题、别名或正文中添加模型厂商、对话工具、导出工具等来源前缀。
- 同一概念只保留一个稳定文件名。重命名时必须同步更新仓库内所有旧链接。
- 避免仅用日期、`新建文档`、`未命名` 等无法表达主题的名称。

## Frontmatter

新增技术笔记应在文件开头使用以下结构：

```yaml
---
title: 清晰的文章标题
type: paper_note
topic: robotics_foundation_model
status: draft
importance: high
updated: YYYY-MM-DD
tags:
  - primary-topic
  - secondary-topic
---
```

- `type` 常用值为 `paper_note` 或 `concept_note`。
- `topic` 使用稳定、简短的英文 snake_case 分类。
- `status` 使用 `draft` 或 `mature`。
- 只有应作为该主题主要参考资料的笔记才使用 `importance: high`。
- `updated` 使用实际编辑日期，格式为 `YYYY-MM-DD`。
- 标签使用小写英文和连字符，避免创建含义重复的标签。
- 修改旧文章时补齐缺失字段，但不要为了统一格式而大范围改写无关正文。

## 正文编辑

- 中文为默认说明语言；模型名、算法名和必要术语可保留英文。
- 新文章只保留一个主标题 `#`，正文层级使用 `##`、`###`，保证 Obsidian 大纲可以稳定导航。
- 保留用户原有技术推导、公式、代码和个人理解，除非用户明确要求重写或校订。
- 公式优先使用 `$...$` 和独立的 `$$ ... $$`，不要改成只在某个网页渲染器中有效的格式。
- 区分论文原文结论、仓库已有解释和新增推断；不能确认的内容不要写成论文事实。
- 清理与知识内容无关的对话导出页脚、分享链接和工具宣传文字。

## 建立文章关系

每篇新增文章至少要连接到主题索引，并与最相关的已有文章建立语义链接。

1. 在正文合适位置或文末的“相关笔记”部分加入 2 至 5 个高相关双链。
2. 内部关系使用 Obsidian wikilink：`[[文件名|显示名称]]`。
3. 通常省略 `.md`；目标文件名在仓库中必须唯一且真实存在。
4. 若文件名可能冲突，使用从仓库根目录开始的路径，例如 `[[RL/PPO|PPO]]`。
5. 只建立可以说明技术继承、概念依赖、方法对照或共同主题的链接，不要为了增加图谱连线而链接无关文章。
6. 对强关系补充反向链接。例如新文章把某模型作为直接后继或重要对照时，也应在旧文章的“相关笔记”中加入新文章。
7. 重命名后搜索旧 basename，并把所有 `[[旧名称...]]` 和 Markdown 路径更新为新名称。

## 更新索引

新增、移动、重命名或大幅修改文章后，按需更新以下文件：

- `index/robotics_papers.md`：机器人、VLA、AWM、机器人基础模型和动作建模文章。
- `index/ai_fundamentals.md`：生成模型、RL、蒸馏、数学和跨主题基础概念。
- `index/master_index.md`：主题范围、新类别和顶层入口发生变化时更新。
- `reading_status.md`：加入已完成或待读列表，并更新顶部日期。
- `README.md`：用户可见的仓库介绍、阅读路线或文章清单发生变化时更新。
- `README_for_GPT.md`：目录结构、主题范围或面向 LLM 的导航规则发生变化时更新。

主题索引中的文章条目保持以下格式：

```markdown
- [[文件名|显示标题]]
  - Topic: 可搜索的核心概念和关键词。
  - Importance: high
  - Notes: 说明何时优先阅读这篇笔记。
```

README 面向 GitHub 阅读，内部文件入口使用带完整相对路径的标准 Markdown 链接：

```markdown
[RT-1](<Robot/VLA/RT-1 论文综述.md>)
```

路径含空格或中文时用尖括号包裹目标。不要把 README 中的链接缩短为只有 basename，否则 GitHub 无法从仓库根目录找到子目录文件。

## 修改后的检查

完成编辑后必须检查：

1. `rg -n "旧文件名" -g "*.md"` 不再返回重命名前的链接。
2. 所有新增或改动的 `[[wikilink]]` 都能解析到唯一的 Markdown 文件。
3. README 中每个本地 `.md` 链接都能从仓库根目录解析到真实文件。
4. 每篇领域文章至少被一个主题索引或 `reading_status.md` 引用。
5. 新文章至少有一个来自其他文章或索引的入链，不应成为关系图中的孤立节点。
6. `git diff --check` 通过。
7. 再次运行 `git status -sb`，确认没有遗漏文件，也没有误改 `.obsidian/` 或无关内容。

如果发现断链，先修复链接和索引，再结束任务。不要仅报告断链数量而留下未处理问题。

## Git 规则

- 普通的“更新笔记”请求只修改和校验本地文件，不自动提交或推送。
- 只有用户明确要求时才执行 `git add`、`git commit` 或 `git push`。
- 提交时将文章重命名、正文链接、索引和导航更新放在同一个有意义的提交中。
- 推送前确认当前分支及远端状态，避免覆盖用户尚未同步的工作。

## 隐私

个人身份、账号、合同、家庭、凭据或其他敏感资料默认不加入面向 LLM 的索引。只有用户明确同意时，才把此类文件加入 `README.md`、`README_for_GPT.md` 或 `index/`。
